## 2. 전략 결정 과정

### 검토한 방법 3가지
| 순위 | 방법 | 장점 | 단점 | 결정 |
|------|------|------|------|------|
| 1 | **YOLOX 재학습 (동공 데이터)** | 기존 파이프라인 100% 재사용, int8 최적화 완료 | 데이터셋 필요, 학습 시간 | **채택** |
| 2 | MediaPipe Iris TFLite → NPU | 사전학습 완료 | 2단계 파이프라인(눈검출→iris), float32만, 복잡 | 테스트만 |
| 3 | 커스텀 경량 모델 | 크기 최적화 가능 | 처음부터 설계 | 보류 |

### 채택 이유: YOLOX 재학습
- 현재 DK 보드의 파이프라인 (카메라→YOLOX→후처리→출력) 을 **그대로 유지**
- 클래스만 "person" → "pupil"로 변경
- 후처리 코드 수정 최소화 (bbox 좌표 기반 PD 계산만 추가)
- int8 양자화 + NPU 최적화가 이미 검증된 경로

## 4. stedgeai 변환 테스트 결과

### 4-1. 기존 YOLOX 모델 변환 (성공)
```bash
stedgeai generate \
  --model st_yolo_x_nano_480_1.0_0.25_3_st_int8.tflite \
  --target stm32n6 \
  --st-neural-art default@user_neuralart_STM32N6570-DK.json \
  --input-data-type uint8 --output-data-type int8
```
- 결과: 성공 (38초)
- 메모리: weights 1.15MB (octoFlash), activations 2.53MB (NPU RAM)

### 4-2. MediaPipe Iris 모델 변환 (성공, 단 float32)
```bash
stedgeai generate \
  --model iris_landmark.tflite \
  --target stm32n6 \
  --st-neural-art default@user_neuralart_STM32N6570-DK.json \
  --input-data-type float32 --output-data-type float32
```
- 결과: 성공 (13초), 단 모든 레이어에 "WARNING: not quantized" 발생
- 입력: float32, 64x64x3 (48KB)
- 출력: float32, 1x213 (iris landmarks) + 1x15 (eye contour)
- 메모리: weights 2.50MB, activations 896KB
- **결론**: NPU에서 돌아가긴 하지만, float32라 성능 최적이 아님
- **한계**: 2단계 파이프라인 (눈 크롭 → iris landmark) 이라 임베디드에서 복잡

→ 이 테스트로 "MediaPipe Iris는 복잡하다"는 결론을 내리고 YOLOX 재학습 방향으로 확정

## 5. 학습 설정 파일

### 파일 경로
`object_detection/config_file_examples/training_pupil_yolox_darknet.yaml`

### 최종 설정 (수정 반영)
| 항목 | 값 | 이유/비고 |
|------|-----|------|
| model_type | st_yoloxn | 현재 DK 보드와 동일 모델 구조 |
| input_shape | (480,480,3) | DK 보드 기존 설정과 동일 |
| class_names | [pupil] | 동공 1클래스 |
| dataset_name | darknet_yolo | 레지스트리 등록명 (pupil_detection 아님!) |
| operation_mode | training | 먼저 학습만 (chain_tqe는 test_path에 .tfs 필요) |
| batch_size | 16 | RTX 4060 8GB VRAM에 맞춤 (CPU 학습 시에도 OK) |
| epochs | 200 | EarlyStopping(patience=40) 포함 |
| validation_split | 0.1 | train에서 10% 분리 (validation_path 사용 안 함) |
| data_dir | .../datasets/pupil/train | **절대경로 필수** (hydra가 CWD 변경함) |
| quantization_path | .../datasets/pupil/train | PTQ 양자화 시 사용할 데이터 |

### 설정 시 주의사항 (삽질 기록)
1. **경로는 반드시 절대경로** - hydra가 작업 디렉토리를 `experiments_outputs/` 하위로 바꿈
2. **dataset_name은 레지스트리 이름** - `darknet_yolo`여야 함 (자유 이름 아님)
3. **data_dir에 이미지가 직접 있어야 함** - 서브폴더만 있는 루트 폴더는 안 됨
4. **validation_path 대신 validation_split 사용** - valid 폴더에 .tfs 파일이 없으면 에러
5. **chain_tqe 모드는 test_path에도 .tfs 필요** - training 모드로 먼저 돌리기

## 9. 트러블슈팅 기록

### Python 3.13 호환성 문제
- **증상**: venv를 Python 3.13으로 만들면 tensorflow 2.18.0 설치 안 됨
- **원인**: TF 2.18은 Python 3.12까지만 지원
- **해결**: Python 3.12.1로 venv 재생성
  ```bash
  /c/Users/USER/AppData/Local/Programs/Python/Python312/python.exe -m venv venv
  ```

### optree 0.19.0 Segfault (핵심!)
- **증상**: keras import 시 Windows access violation (segfault)
- **원인**: optree 0.19.0의 네이티브 모듈이 Windows Python 3.12에서 충돌
- **에러 위치**: `optree/accessors.py` → `_C.pyd` 로딩 시
- **해결**: `pip install optree==0.13.1` 로 다운그레이드
- **주의**: keras가 optree를 의존하므로, 다른 패키지가 optree를 업그레이드하면 다시 발생

### clearml + torch DLL 충돌
- **증상**: clearml import 후 torch import 시 `c10.dll` 로딩 실패 (WinError 1114)
- **원인**: clearml이 DLL 로딩 환경을 변경하여 torch 네이티브 라이브러리 충돌
- **해결**: `stm32ai_main.py`에서 `import torch`를 `from clearml import Task` 보다 앞에 배치
- **수정 파일**: `object_detection/stm32ai_main.py` (line 21~26)

### pesq 빌드 실패
- **증상**: `Failed to build pesq` - C 컴파일러 (Visual Studio Build Tools) 없음
- **영향**: speech_enhancement 모듈에서만 사용, object_detection에는 불필요
- **해결**: 더미 모듈 생성 (`venv\Lib\site-packages\pesq\__init__.py`)

### pycocotools 빌드 실패
- **증상**: `Failed to build pycocotools` - C 컴파일러 없음
- **해결**: requirements.txt에서 빼고 설치 후, pip install pycocotools 별도 실행 시 성공
  (캐시된 wheel이 있어서 두 번째 시도에 성공한 것으로 추정)

### TensorFlow GPU 미감지
- **증상**: `tf.config.list_physical_devices('GPU')` → 빈 리스트
- **원인**: CUDA Toolkit 미설치 (nvidia-smi는 있으나 nvcc 없음)
- **영향**: CPU로 학습 진행 (느리지만 가능)
- **해결 방안**: CUDA 12.3 + cuDNN 설치 시 GPU 학습 가능 (다음 세션)

### darknet_yolo 데이터셋 로딩 에러들
- **data_dir 상대경로 문제**: hydra가 CWD를 바꿔서 상대경로 안 됨 → 절대경로로 해결
- **dataset_name 레지스트리 미스매치**: `pupil_detection` → `darknet_yolo`로 수정
- **data_dir에 이미지 없음**: 루트 폴더가 아닌 train 폴더를 지정
- **validation_path에 .tfs 없음**: validation_path 대신 validation_split: 0.1 사용
- **test_path에 .tfs 없음**: chain_tqe → training 모드로 변경하여 우회

### matplotlib 창 블로킹
- **증상**: 학습 중 matplotlib 그래프 창이 떠서 프로세스가 멈춤
- **원인**: `display_figures: false`만으로는 모든 곳을 커버 못함
- **해결**: `MPLBACKEND=Agg` 환경변수로 비대화형 백엔드 강제 설정

### Windows TensorFlow GPU 미지원 (TF 2.11+)
- **증상**: CUDA pip 패키지 설치해도 `Built with CUDA: False`, GPU 리스트 빈 값
- **원인**: TF 2.11부터 Windows native pip 패키지에 CUDA 지원이 빌드에서 제외됨
- **확인**: `python -c "import tensorflow as tf; print(tf.test.is_built_with_cuda())"` → False
- **시도한 것**: `pip install nvidia-cudnn-cu12 nvidia-cuda-runtime-cu12` → 설치는 되지만 효과 없음
- **대안**:
  1. WSL2 (Windows Subsystem for Linux)에서 Linux TF GPU 사용 — 환경 구축 복잡
  2. **Google Colab (채택)** — 무료 T4 GPU, 브라우저에서 바로 실행
  3. tensorflow-directml — Python 3.12에서 배포판 없음
