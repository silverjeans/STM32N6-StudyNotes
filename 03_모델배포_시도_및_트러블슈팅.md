## 5. 보드 배포 결과 및 실패 원인

### 근본 원인: 모델 구조 불일치 (앵커 수)

| | Person 모델 (동작함) | Pupil 모델 (실패) |
|--|---------------------|-------------------|
| **앵커 수** | **3** | **1** |
| **출력 채널** | **18** (3×6) | **6** (1×6) |
| **파라미터** | 1,139,414 | 889,394 |
| **후처리** | 3앵커 기준 디코딩 | 1앵커 → 후처리와 안 맞음 |

**왜 1앵커가 됐나?**
- ST Model Zoo의 `st_yoloxn` 기본 설정에서 `yolo_anchors`를 지정 안 하면 `[0.5, 0.5]` (1쌍 = 1앵커)가 기본값
- Person 모델은 ST가 3앵커로 사전학습한 것이라 `yolo_anchors`가 6개 값 (3쌍 = 3앵커)
- Colab 학습 config에서 `yolo_anchors`를 명시하지 않아서 1앵커로 학습됨

**교훈**: 기존 모델의 출력 형식을 반드시 확인하고, 재학습 시 동일한 앵커 구조를 사용해야 함

## 6. MediaPipe Iris 시도 (실패)

### 시도한 이유
- YOLOX 앵커 불일치 문제를 우회하기 위해
- 이전에 PC 웹캠에서 동공 검출 성공 경험
- stedgeai NPU 변환도 성공했었음

### 수정한 파일
- `app_config.h`: YOLOX 설정 → Iris landmark 설정
- `main.c`: YOLOX 후처리 → Iris 좌표 추출 + 십자선 표시 + UART 출력
- `app_postprocess.h`: `POSTPROCESS_IRIS_LANDMARK` 정의 추가

### 1차 실패: HardFault (입력 64x64)
- stai_network.h에서 입력 64x64x3 → DCMIPP가 64x64로 캡처 시도
- DCMIPP 하드웨어의 최소 출력 크기 제한에 걸려 HardFault

### 2차 실패: HardFault (480→64 소프트웨어 리사이즈)
- 카메라는 480x480 캡처, 소프트웨어로 64x64 리사이즈 후 NPU 입력
- stai_network.h의 크기를 속이거나 DCMIPP만 별도 설정 시도
- NPU 내부 버퍼 크기 불일치로 추론 중 HardFault (`forward_lite_com2d_ff32of32w32_group`)

### 결론
- MediaPipe Iris를 이 프로젝트에 넣으려면 DCMIPP, NPU 버퍼, float32 변환 등 **깊은 수준의 수정**이 필요
- 현 시점에서는 비현실적 → YOLOX 3앵커 재학습으로 방향 전환

## 9. 오늘의 교훈

### 1. 모델 출력 형식은 반드시 기존과 일치시켜야 한다
- 후처리 코드가 특정 출력 형식(앵커 수, 채널 수, 데이터 타입)에 맞춰져 있음
- 새 모델 학습 시 기존 모델의 출력 구조를 먼저 확인하고 동일하게 설정

### 2. stedgeai 출력 데이터 타입 주의
- person 모델: float32 출력
- 양자화 모델: int8 출력 가능 → stedgeai에서 `--output-data-type float32`로 맞춰야 함

### 3. DCMIPP는 너무 작은 해상도를 지원 안 함
- 64x64 같은 작은 입력은 DCMIPP에서 HardFault 발생
- 소프트웨어 리사이즈를 하려면 NPU 버퍼 관리가 복잡해짐

### 4. stai_network.h를 임의로 수정하면 NPU 크래시
- stedgeai가 생성한 값은 NPU 하드웨어 설정과 밀접하게 연결
- 입력 크기를 속이면 NPU 내부 메모리 접근 위반 발생

### 5. Windows TF 2.18은 GPU를 지원하지 않는다
- TF 2.11부터 Windows native에서 CUDA GPU 빌드 제외
- 해결: WSL2 또는 Google Colab 사용

## 12. WSL2 설치 (로컬 GPU 학습 환경)

### 왜 WSL2가 필요한가?
- **Windows TF 2.18은 GPU를 지원하지 않음** (TF 2.11부터 Windows에서 CUDA 빌드 제외)
- Colab 무료는 GPU 사용량 제한 있음 (오늘 소진됨)
- **WSL2 (Windows Subsystem for Linux)** = Windows 안에서 Linux 실행
- Linux TF는 GPU 지원 → **RTX 4060 Laptop을 로컬에서 사용 가능**
- 한 번 세팅하면 이후로 GPU 학습이 자유로움

### 설치 과정

#### 1. WSL 설치
- PowerShell (관리자) 에서: `wsl --install`
- PC 재부팅 필요

#### 2. Ubuntu 초기 설정
- Windows 검색 → "Ubuntu" 실행
- 사용자명: `molly`
- 비밀번호 설정 실패 시: PowerShell(관리자)에서 `wsl -u root passwd molly` 로 설정
- Linux에서 비밀번호 입력 시 **화면에 아무것도 안 보이는 게 정상** (보안 때문)

#### 3. 패키지 설치 (진행 중)
```bash
sudo apt update && sudo apt install -y python3 python3-pip python3-venv
```

### WSL2 환경 구축 완료!

#### Python 가상환경
- 경로: `~/tf_gpu/`
- 활성화: `source ~/tf_gpu/bin/activate`
- TensorFlow: 2.21.0 (CUDA 13.2 호환)
- GPU: RTX 4060 Laptop GPU 인식 완료

#### NVIDIA 드라이버 업데이트
- 기존: 546.33 (CUDA 12.3) → TF 2.18과 충돌
- 업데이트: **595.97 (CUDA 13.2)** → TF 2.21.0과 호환
- TF 2.18은 CUDA 12.5 런타임 사용 → 드라이버 546은 12.3까지만 지원 → 크래시
- TF 2.21.0으로 업그레이드하여 해결

#### 트러블슈팅
- `cudaGetDevice() failed: CUDA driver version is insufficient` → 드라이버 업데이트로 해결
- `gpu_memory_limit` 설정 시 에러 → config에서 비활성화
- `torch` import 후 `clearml` import하면 DLL 충돌 → import 순서 수정
- TF 2.18 + CUDA 13.2 = XLA 커널 크래시 → TF 2.21.0 업그레이드로 해결

### 현재 학습 진행 중 (WSL2 로컬 GPU)
- 데이터: 흑백 pupil 12,138장 + 컬러 Unity Eyes 8,100장 = 20,238장
- 모델: YOLOX 3앵커 (person 모델 호환)

### WSL2 관련 명령어
